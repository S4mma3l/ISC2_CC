Un ejemplo es 00-13-02-1F-58-F5. los primeros 3 bytes (24 bits) de la dirección indican el proveedor o fabricante de la interfaz de red física. Dos dispositivos no pueden tener la misma dirección MAC en la misma red local; de lo contrario, se produce un conflicto de direcciones.

Los ejemplos son 192.168.1.1 y 2001:db8::ffff:0:1.

Modelo de interconexión de sistemas abiertos (OSI)

El Modelo OSI fue desarrollado para establecer una forma común de describir la estructura de comunicación para sistemas informáticos interconectados. El modelo OSI sirve como marco abstracto, o modelo teórico, sobre cómo deberían funcionar los protocolos en un mundo ideal, en un hardware ideal. Por lo tanto, el modelo OSI se ha convertido en una referencia conceptual común que se utiliza para comprender la comunicación de varios componentes jerárquicos desde las interfaces de software hasta el hardware físico.

El modelo OSI divide las tareas de red en siete capas distintas. Cada capa es responsable de realizar tareas u operaciones específicas con el objetivo de respaldar el intercambio de datos (en otras palabras, la comunicación en red) entre dos computadoras. Las capas se referencian indistintamente por nombre o número de capa. Por ejemplo, la capa 3 también se conoce como capa de red. Las capas están ordenadas específicamente para indicar cómo fluye la información a través de los distintos niveles de comunicación. Cada capa se comunica directamente con la capa superior y la capa inferior. Por ejemplo, la capa 3 se comunica con las capas de enlace de datos (2) y transporte (4).

Las capas de aplicación, presentación y sesión (5-7) se conocen comúnmente como datos. Sin embargo, cada capa tiene el potencial para realizar la encapsulación. La encapsulación es la adición de datos de encabezado y posiblemente de pie de página (remolque) mediante un protocolo utilizado en esa capa del modelo OSI. La encapsulación es particularmente importante cuando se analizan las capas de transporte, red y enlace de datos (2 a 4), que generalmente incluyen algún tipo de encabezado. En la capa física (1), la unidad de datos se convierte en binario, es decir, 01010111, y se envía a través de cables físicos, como un cable Ethernet.

El Modelo OSI fue desarrollado para establecer una forma común de describir la estructura de comunicación para sistemas informáticos interconectados. El modelo OSI sirve como marco abstracto, o modelo teórico, sobre cómo deberían funcionar los protocolos en un mundo ideal, en un hardware ideal. Por lo tanto, el modelo OSI se ha convertido en una referencia conceptual común que se utiliza para comprender la comunicación de varios componentes jerárquicos desde las interfaces de software hasta el hardware físico.



El modelo OSI divide las tareas de red en siete capas distintas. Cada capa es responsable de realizar tareas u operaciones específicas con el objetivo de respaldar el intercambio de datos (en otras palabras, la comunicación en red) entre dos computadoras. Las capas se referencian indistintamente por nombre o número de capa. Por ejemplo, la capa 3 también se conoce como capa de red. Las capas están ordenadas específicamente para indicar cómo fluye la información a través de los distintos niveles de comunicación. Cada capa se comunica directamente con la capa superior y la capa inferior. Por ejemplo, la capa 3 se comunica con las capas de enlace de datos (2) y transporte (4).



Las capas de aplicación, presentación y sesión (5-7) se conocen comúnmente como datos. Sin embargo, cada capa tiene el potencial para realizar la encapsulación. La encapsulación es la adición de datos de encabezado y posiblemente de pie de página (remolque) mediante un protocolo utilizado en esa capa del modelo OSI. La encapsulación es particularmente importante cuando se analizan las capas de transporte, red y enlace de datos (2 a 4), que generalmente incluyen algún tipo de encabezado. En la capa física (1), la unidad de datos se convierte en binario, es decir, 01010111, y se envía a través de cables físicos, como un cable Ethernet.

Protocolo de Internet (IPv4 e IPv6)

Actualmente, IP se implementa y utiliza en todo el mundo en dos versiones principales. IPv4 proporciona un espacio de direcciones de 32 bits, que a fines de la década de 1980 se proyectó que estaría agotado. IPv6 se introdujo en diciembre de 1995 y proporciona un espacio de direcciones de 128 bits junto con otras funciones importantes

Los hosts/dispositivos IP asocian una dirección con una dirección lógica única. Una dirección IPv4 se expresa como cuatro octetos separados por un punto (.), por ejemplo, 216.12.146.140. Cada octeto puede tener un valor entre 0 y 255. Sin embargo, 0 es la red en sí (no un dispositivo en esa red) y 255 generalmente se reserva para fines de transmisión. Cada dirección se subdivide en dos partes: el número de red y el host. El número de red asignado por una organización externa, como la Corporación de Internet para la Asignación de Nombres y Números (ICANN), representa la red de la organización. El host representa la interfaz de red dentro de la red.

Para facilitar la administración de la red, las redes normalmente se dividen en subredes. Debido a que las subredes no se pueden distinguir con el esquema de direccionamiento analizado hasta ahora, se usa un mecanismo separado, la máscara de subred, para definir la parte de la dirección que se usa para la subred. La máscara generalmente se convierte a notación decimal como 255.255.255.0.

Con el número cada vez mayor de computadoras y dispositivos en red, está claro que IPv4 no proporciona suficientes direcciones para nuestras necesidades. Para superar esta deficiencia, IPv4 se subdividió en rangos de direcciones públicas y privadas. Las direcciones públicas están limitadas con IPv4, pero este problema se solucionó en parte con el direccionamiento privado. Cualquiera puede compartir las direcciones privadas y es muy probable que todos en su calle estén usando el mismo esquema de direcciones.

¿Qué es Wi-Fi?

Las redes inalámbricas son un método popular para conectar sistemas corporativos y domésticos debido a la facilidad de implementación y al costo relativamente bajo. Ha hecho que las redes sean más versátiles que nunca. Las estaciones de trabajo y los sistemas portátiles ya no están atados a un cable, sino que pueden moverse libremente dentro del rango de señal de los puntos de acceso inalámbrico implementados. Sin embargo, con esta libertad vienen vulnerabilidades adicionales.

El rango de Wi-Fi es generalmente lo suficientemente amplio para la mayoría de los hogares u oficinas pequeñas, y los extensores de rango se pueden colocar estratégicamente para extender la señal a campus u hogares más grandes. Con el tiempo, el estándar Wi-Fi ha evolucionado, y cada versión actualizada es más rápida que la anterior.

En una LAN, los actores de amenazas deben ingresar al espacio físico o las inmediaciones del propio medio físico. Para redes cableadas, esto se puede hacer colocando toques de rastreo en los cables, conectando dispositivos USB o usando otras herramientas que requieren acceso físico a la red. En la capa de aplicación del modelo TCP/IP (que incluye las capas de sesión, presentación y aplicación del modelo OSI) residen numerosos protocolos específicos de aplicaciones o servicios. Los tipos de datos se asignan mediante números de puerto asociados con los servicios. Por ejemplo, el tráfico web (o HTTP) es el puerto 80. El tráfico web seguro (o HTTPS) es el puerto 443.     Puertos registrados (1024–49151): estos puertos a menudo se asocian con aplicaciones propietarias de proveedores y desarrolladores. Los ejemplos incluyen la autenticación del servicio de usuario de acceso telefónico de autenticación remota (RADIUS) (1812), Microsoft SQL Server (1433/1434) y la API REST de Docker (2375/2376).
    Puertos privados o dinámicos (49152–65535): siempre que se solicite un servicio que esté asociado con puertos bien conocidos o registrados, esos servicios responderán con un puerto dinámico que se usa para esa sesión y luego se libera.

## Resumen
Sistema de detección de intrusos (IDS)

Una intrusión ocurre cuando un atacante puede eludir o frustrar los mecanismos de seguridad y obtener acceso a los recursos de una organización. La detección de intrusos es una forma específica de monitoreo que monitorea la información registrada y los eventos en tiempo real para detectar actividad anormal que indique un posible incidente o intrusión. Un sistema de detección de intrusiones (IDS) automatiza la inspección de registros y eventos del sistema en tiempo real para detectar intentos de intrusión y fallas del sistema. Un IDS está pensado como parte de un plan de seguridad de defensa en profundidad. Funcionará con otros mecanismos de seguridad, como los cortafuegos, y los complementará, pero no los reemplaza.

Los IDS pueden reconocer ataques que provienen de conexiones externas, como un ataque de Internet, y ataques que se propagan internamente, como un gusano malicioso. Una vez que detectan un evento sospechoso, responden enviando alertas o activando alarmas. Un objetivo principal de un IDS es proporcionar un medio para una respuesta oportuna y precisa a las intrusiones.

La detección y prevención de intrusiones se refieren a las capacidades que forman parte del aislamiento y la protección de un dominio o una zona más segura o de mayor confianza de otra que es menos confiable o menos segura. Estas son funciones naturales que se esperan de un firewall, por ejemplo. 

Los tipos de IDS se clasifican comúnmente como basados en host y basados en red. Un IDS basado en host (HIDS) supervisa una sola computadora o host. Un IDS basado en red (NIDS) supervisa una red observando los patrones de tráfico de la red.

Sistema de detección de intrusos basado en host (HIDS)

Un HIDS monitorea la actividad en una sola computadora, incluidas las llamadas de proceso y la información registrada en los registros del sistema, la aplicación, la seguridad y el firewall basado en host. A menudo, puede examinar eventos con más detalle que un NIDS y puede identificar archivos específicos comprometidos en un ataque. También puede rastrear los procesos empleados por el atacante. Una ventaja de los HIDS sobre los NIDS es que los HIDS pueden detectar anomalías en el sistema host que los NIDS no pueden detectar. Por ejemplo, un HIDS puede detectar infecciones en las que un intruso se ha infiltrado en un sistema y lo está controlando de forma remota. Los HIDS son más costosos de administrar que los NIDS porque requieren atención administrativa en cada sistema, mientras que los NIDS generalmente admiten la administración centralizada. Un HIDS no puede detectar ataques de red en otros sistemas.

Sistema de detección de intrusos en la red (NIDS)

Un NIDS monitorea y evalúa la actividad de la red para detectar ataques o anomalías en los eventos. No puede monitorear el contenido del tráfico cifrado, pero puede monitorear otros detalles del paquete. Un solo NIDS puede monitorear una gran red mediante el uso de sensores remotos para recopilar datos en ubicaciones clave de la red que envían datos a una consola de administración central. Estos sensores pueden monitorear el tráfico en enrutadores, firewalls, conmutadores de red que admiten la duplicación de puertos y otros tipos de derivaciones de red. Un NIDS tiene un efecto negativo muy pequeño en el rendimiento general de la red y, cuando se implementa en un sistema de propósito único, no afecta negativamente el rendimiento en ninguna otra computadora. Un NIDS generalmente puede detectar el inicio de un ataque o ataques en curso, pero no siempre puede proporcionar información sobre el éxito de un ataque. No sabrán si un ataque afectó sistemas específicos, cuentas de usuario, archivos o aplicaciones.

## Resumen
Cortafuegos

En la construcción de edificios o el diseño de vehículos, un cortafuegos es una barrera física construida especialmente que evita la propagación del fuego de un área de la estructura a otra o de un compartimento de un vehículo a otro. ## Resumen
Memorando de Entendimiento (MOU)/Memorándum de Acuerdo (MOA)

Algunas organizaciones que buscan minimizar el tiempo de inactividad y mejorar las capacidades de BC (Continuidad del negocio) y DR (Recuperación ante desastres) crearán acuerdos con otras organizaciones similares. Acuerdan que si una de las partes experimenta una emergencia y no puede operar dentro de sus propias instalaciones, la otra parte compartirá sus recursos y les permitirá operar dentro de las suyas para mantener funciones críticas. Estos acuerdos a menudo incluso incluyen competidores, porque sus instalaciones y recursos satisfacen las necesidades de su industria en particular.

Por ejemplo, el Hospital A y el Hospital B son competidores en la misma ciudad. Los hospitales crean un acuerdo entre ellos: si algo malo le sucede al Hospital A (incendio, inundación, amenaza de bomba, corte de energía, etc.), ese hospital puede enviar temporalmente personal y sistemas para trabajar dentro del Hospital B para quedarse. en el negocio durante la interrupción (y el Hospital B puede reubicarse en el Hospital A, si el Hospital B tiene un problema similar). Los hospitales han decidido que no van a competir en base a la seguridad y la protección, van a competir en servicio, precio y lealtad del cliente. De esta manera, se protegen a sí mismos y a la industria de la salud en su conjunto.

Estos acuerdos se denominan acuerdos operativos conjuntos (JOA) o memorandos de entendimiento (MOU) o memorandos de acuerdo (MOA). A veces, estos acuerdos son obligatorios por requisitos reglamentarios, o pueden ser simplemente parte de las salvaguardas administrativas instituidas por una entidad dentro de las pautas de su industria.

Algunas organizaciones que buscan minimizar el tiempo de inactividad y mejorar las capacidades de BC (Continuidad del negocio) y DR (Recuperación ante desastres) crearán acuerdos con otras organizaciones similares. Acuerdan que si una de las partes experimenta una emergencia y no puede operar dentro de sus propias instalaciones, la otra parte compartirá sus recursos y les permitirá operar dentro de las suyas para mantener funciones críticas. Estos acuerdos a menudo incluso incluyen competidores, porque sus instalaciones y recursos satisfacen las necesidades de su industria en particular.



Por ejemplo, el Hospital A y el Hospital B son competidores en la misma ciudad. Los hospitales crean un acuerdo entre ellos: si algo malo le sucede al Hospital A (incendio, inundación, amenaza de bomba, corte de energía, etc.), ese hospital puede enviar temporalmente personal y sistemas para trabajar dentro del Hospital B para quedarse. en el negocio durante la interrupción (y el Hospital B puede reubicarse en el Hospital A, si el Hospital B tiene un problema similar). Los hospitales han decidido que no van a competir en base a la seguridad y la protección, van a competir en servicio, precio y lealtad del cliente. De esta manera, se protegen a sí mismos y a la industria de la salud en su conjunto.



Estos acuerdos se denominan acuerdos operativos conjuntos (JOA) o memorandos de entendimiento (MOU) o memorandos de acuerdo (MOA). A veces, estos acuerdos son obligatorios por requisitos reglamentarios, o pueden ser simplemente parte de las salvaguardas administrativas instituidas por una entidad dentro de las pautas de su industria.



La diferencia entre un MOA o MOU y un SLA es que un Memorando de Entendimiento está más directamente relacionado con lo que se puede hacer con un sistema o la información.



El acuerdo de nivel de servicio se reduce al nivel granular. Con la computación en la nube, necesito tener acceso a la información en mis sistemas de respaldo en 10 minutos. Un SLA especifica los aspectos más complejos de los servicios.

Si el SLA promete 100 por ciento de accesibilidad a la información, ¿el acceso es directo a usted en este momento, o es acceso a su sitio web oa través de su portal cuando abran el lunes? Ahí es donde confiará en su equipo legal, que puede supervisar y revisar las condiciones cuidadosamente antes de firmar la línea punteada en la parte inferior.?

La computación en la nube generalmente se asocia con un conjunto de recursos informáticos basados en Internet y, por lo general, se vende como un servicio proporcionado por un proveedor de servicios en la nube (CSP).

La computación en la nube es muy similar a la red eléctrica o de energía. Se aprovisiona en una ubicación geográfica y se obtiene mediante un medio eléctrico que no es necesariamente obvio para el consumidor. Existen varias definiciones de lo que significa la computación en la nube de acuerdo con los principales estándares, incluido el NIST. Esta definición del NIST se usa comúnmente en todo el mundo, citada por profesionales y otros para aclarar qué significa el término "nube":

“un modelo para permitir el acceso a la red ubicuo, conveniente y bajo demanda a un conjunto compartido de recursos informáticos configurables (como redes, servidores, almacenamiento, aplicaciones y servicios) que se pueden aprovisionar y liberar rápidamente con un mínimo esfuerzo de gestión o proveedor de servicios. Interacción." NIST SP 800-145

    Permite a una empresa escalar software nuevo o servicios/soluciones basados en datos a través de sistemas en la nube rápidamente y sin tener que instalar hardware masivo localmente.

Plataforma como servicio (PaaS)
Plataforma como servicio (PaaS): una nube proporciona un entorno para que los clientes lo utilicen para crear y operar su propio software. PaaS es una forma para que los clientes alquilen hardware, sistemas operativos, almacenamiento y capacidad de red a través de Internet de un proveedor de servicios en la nube. El modelo de prestación de servicios permite a los clientes alquilar servidores virtualizados y servicios asociados para ejecutar aplicaciones existentes o desarrollar y probar otras nuevas. El consumidor no administra ni controla la infraestructura de la nube subyacente, incluida la red, los servidores, los sistemas operativos o el almacenamiento, pero tiene control sobre las aplicaciones implementadas y, posiblemente, las configuraciones del entorno de alojamiento de aplicaciones. Una nube PaaS proporciona un conjunto de herramientas para desarrollar, implementar y administrar convenientemente software de aplicación que está estructurado para admitir una gran cantidad de consumidores, procesar grandes cantidades de datos y potencialmente acceder desde cualquier punto de Internet. Las nubes PaaS generalmente proporcionarán un conjunto de componentes básicos de software y un conjunto de herramientas de desarrollo, como lenguajes de programación y entornos de tiempo de ejecución compatibles que facilitan la construcción de aplicaciones escalables y de alta calidad. Además, las nubes PaaS normalmente proporcionarán herramientas que ayuden con la implementación de nuevas aplicaciones. En algunos casos, implementar una nueva aplicación de software en una nube PaaS no es mucho más difícil que cargar un archivo en un servidor web. Las nubes PaaS generalmente también proporcionarán y mantendrán los recursos informáticos (por ejemplo, procesamiento, almacenamiento y redes) que las aplicaciones de los consumidores necesitan para operar. Las nubes PaaS brindan muchos beneficios para los desarrolladores, incluido que el sistema operativo se puede cambiar y actualizar con frecuencia, junto con las funciones asociadas y los servicios del sistema.

Confianza cero

Las redes de confianza cero suelen ser redes microsegmentadas, con cortafuegos en casi todos los puntos de conexión. La confianza cero encapsula los activos de información, los servicios que se aplican a ellos y sus propiedades de seguridad. Este concepto reconoce que una vez dentro de un entorno de confianza pero verificación, un usuario tiene capacidades quizás ilimitadas para desplazarse, identificar activos y sistemas y, potencialmente, encontrar vulnerabilidades explotables. Colocar una mayor cantidad de firewalls u otros dispositivos de control de límites de seguridad en toda la red aumenta la cantidad de oportunidades para detectar a un alborotador antes de que se produzca el daño. Muchas arquitecturas empresariales están llevando esto al extremo de microsegmentar sus redes internas, lo que impone la reautenticación frecuente de una ID de usuario, como se muestra en esta imagen.

En un entorno de confianza cero, se agregan puntos de control adicionales. Su identidad (boleto) se valida para acceder a los asientos del nivel del piso y nuevamente para acceder al área detrás del escenario. Sus credenciales deben ser válidas en los 3 niveles para conocer a las estrellas del espectáculo.

La confianza cero es un enfoque de diseño en evolución que reconoce que incluso los sistemas de control de acceso más robustos tienen sus puntos débiles. Agrega defensas a nivel de usuario, activo y datos, en lugar de depender de la defensa del perímetro. En extremo, insiste en que todo proceso o acción que intente realizar un usuario debe ser autenticado y autorizado; la ventana de confianza se vuelve cada vez más pequeña.

Mientras que la microsegmentación agrega perímetros internos, la confianza cero se enfoca en los activos o datos, en lugar del perímetro. Zero Trust construye puertas más efectivas para proteger los activos directamente en lugar de construir muros adicionales o más altos.

Mire alrededor de su propio entorno y verá rápidamente la escala de su uso.

Habiendo identificado la necesidad de una solución NAC, necesitamos identificar qué capacidades puede proporcionar una solución. Como sabemos, todo comienza con una política. Las políticas de control de acceso de la organización y las políticas de seguridad asociadas deben aplicarse a través de los dispositivos NAC. Recuerde, por supuesto, que un dispositivo de control de acceso solo aplica una política y no crea una.

## Resumen
Segmentación para Sistemas Embebidos e IoT

Un sistema embebido es una computadora implementada como parte de un sistema más grande. El sistema integrado generalmente se diseña en torno a un conjunto limitado de funciones específicas en relación con el producto más grande del que es un componente. Los ejemplos de sistemas integrados incluyen impresoras conectadas a la red, televisores inteligentes, controles HVAC, electrodomésticos inteligentes, termostatos inteligentes y dispositivos médicos.

Los dispositivos habilitados para red son cualquier tipo de dispositivo portátil o no portátil que tiene capacidades de red nativas. Por lo general, se supone que la red en cuestión es de tipo inalámbrico, normalmente proporcionada por una empresa de telecomunicaciones móviles. Los dispositivos habilitados para red incluyen teléfonos inteligentes, teléfonos móviles, tabletas, televisores inteligentes o reproductores multimedia de transmisión (como Roku Player, Amazon Fire TV o Google Android TV/Chromecast), impresoras conectadas a la red, sistemas de juegos y mucho más.

El Internet de las cosas (IoT) es la colección de dispositivos que pueden comunicarse a través de Internet entre sí o con una consola de control para afectar y monitorear el mundo real. Red de área local virtual (VLAN)

Las redes de área local virtual (VLAN) permiten a los administradores de red usar conmutadores para crear segmentos de LAN basados en software, que pueden segregar o consolidar el tráfico a través de múltiples puertos de conmutador. Los dispositivos que comparten una VLAN se comunican a través de conmutadores como si estuvieran en la misma red de Capa 2. Esta imagen muestra diferentes VLAN (rojo, verde y azul) que conectan conjuntos separados de puertos, mientras comparten el mismo segmento de red (que consta de los dos conmutadores y su conexión). Dado que las VLAN actúan como redes discretas, se deben habilitar las comunicaciones entre las VLAN. El tráfico de difusión se limita a la VLAN, lo que reduce la congestión y reduce la eficacia de algunos ataques. La administración del entorno se simplifica, ya que las VLAN se pueden reconfigurar cuando las personas cambian de ubicación física o necesitan acceder a diferentes servicios. Las VLAN se pueden configurar según el puerto del conmutador, la subred IP, la dirección MAC y los protocolos.

Las VLAN no garantizan la seguridad de una red. A primera vista, puede parecer que el tráfico no se puede interceptar porque la comunicación dentro de una VLAN está restringida a los dispositivos miembros. Sin embargo, existen ataques que permiten a un usuario malintencionado ver el tráfico de otras VLAN (los llamados saltos de VLAN). La tecnología VLAN es solo una herramienta que puede mejorar la seguridad general del entorno de red.

Red privada virtual (VPN)

Una red privada virtual (VPN) no es necesariamente un túnel encriptado. Es simplemente una conexión punto a punto entre dos hosts que les permite comunicarse. Por supuesto, la VPN puede proporcionar comunicaciones seguras, pero solo si los protocolos de seguridad se han seleccionado y configurado correctamente para proporcionar una ruta confiable a través de una red que no es confiable, como Internet. Los usuarios remotos emplean VPN para acceder a la red de su organización y, según la implementación de la VPN, pueden tener a su disposición la mayoría de los mismos recursos que si estuvieran físicamente en la oficina. Como alternativa a las costosas conexiones dedicadas punto a punto, las organizaciones utilizan VPN de puerta a puerta para transmitir información de forma segura a través de Internet entre sitios o incluso con socios comerciales.

## Resumen
En este capítulo, cubrimos las redes de computadoras y la seguridad de la red. Una red es simplemente dos o más computadoras conectadas entre sí para compartir datos, información o recursos. Hay muchos tipos de redes, como LAN, WAN, WLAN y VPN, por nombrar algunas. Algunos de los dispositivos que se encuentran en una red pueden ser concentradores, conmutadores, enrutadores, cortafuegos, servidores, terminales (por ejemplo, computadoras de escritorio, portátiles, tabletas, teléfonos móviles, VOIP o cualquier otro dispositivo de usuario final). Otros términos de red que debe conocer y comprender incluyen puertos, protocolos, ethernet, Wi-Fi, dirección IP y dirección MAC.

Los dos modelos discutidos en este capítulo son OSI y TCP/IP. El modelo OSI tiene siete capas y el TCP/IP cuatro. Ambos toman los 1 y 0 de la capa de interfaz física o de red, donde se conectan los cables o Wi-Fi, a la capa de aplicación, donde los usuarios interactúan con los datos. Los datos atraviesan la red como paquetes, con encabezados o pies de página que se agregan y eliminan según pasan de una capa a otra. Esto ayuda a enrutar los datos y garantiza que los paquetes no se pierdan y permanezcan juntos. IPv4 está siendo eliminado lentamente por IPv6 para mejorar la seguridad, mejorar la calidad del servicio y admitir más dispositivos.

Como se mencionó, Wi-Fi ha reemplazado muchas de nuestras redes cableadas y, con su facilidad de uso, también trae problemas de seguridad. Asegurar el Wi-Fi es muy importante.

Luego aprendimos sobre algunos de los ataques en una red, por ejemplo, ataques DoS/DDoS, ataques de fragmentos, ataques de paquetes de gran tamaño, ataques de suplantación de identidad y ataques de intermediario. También discutimos los puertos y protocolos que conectan la red y los servicios que se utilizan en las redes, desde puertos físicos, por ejemplo, puerto LAN, que conectan los cables, hasta puertos lógicos, por ejemplo, 80 o 443, que conectan los protocolos/servicios.

Luego examinamos algunas posibles amenazas para una red, incluida la suplantación de identidad, DoS/DDoS, virus, gusanos, troyanos, ataques en la ruta (man-in-the-middle) y ataques de canal lateral. El capítulo continuó discutiendo cómo identificar amenazas, por ejemplo, usando IDS/NIDS/HIDS o SIEM, y prevenir amenazas, por ejemplo, usando antivirus, escaneos, firewalls o IPS/NIPS/HIPS. Discutimos los centros de datos locales y sus requisitos, por ejemplo, energía, HVAC, supresión de incendios, redundancia y MOU/MOA. Revisamos la nube y sus características, para incluir modelos de servicio: SaaS, IaaS y PaaS; y modelos de despliegue: público, privado, comunitario e híbrido. También se discutió la importancia de un MSP y un SLA.
